# 项目经历

## Agent 记忆系统（个人项目）

**技术栈：** Python / LangGraph / FAISS / Redis / MCP / FastAPI / DeepSeek-V3 / BAAI Embedding

从零设计并实现了一个具备短期+长期记忆的个人 AI 助手系统。核心要解决的问题是：如何让 Agent 记住跨会话的用户信息、如何在多轮对话中维护上下文、以及如何让记忆系统的能力边界清晰且可扩展。

**设计思路：**

- **Agent 无状态，状态在外部**：Agent 本身不存任何用户数据，所有上下文通过工具调外部服务获取。会话存在 Redis，记忆存在 FAISS。Agent 可随时销毁重建，靠 user_id + session_id 恢复——与 HTTP 无状态同构，天然支持水平扩展

- **决策权交还 Agent**：不写 if/else 规则流控制记忆注入时机。改为提供 is_new_conversation 工具，让 Agent 自主查询 Redis 判断是否新对话，自主决定何时检索记忆。系统只暴露真实世界信息，不替 Agent 做决策

- **工具驱动的渐进式加载**：基础能力直接传给 Agent，扩展能力注册到独立向量库。Agent 通过元工具语义检索按需发现和加载。新增工具只需定义并注册，启动时自动入向量库，提示词不需要改

- **提示词降为身份约束**：工具描述即能力说明书，LLM 从 Function Calling 拿到精确的工具名和参数。提示词只写"你是谁"和"不能做什么"

**核心实现：**

- 三层解耦架构：RAG 记忆服务（FastAPI + FAISS + 本地 BGE Embedding）→ MCP 工具网关（FastMCP SSE）→ Agent 推理引擎（LangGraph ReAct）。Agent 和外部 MCP 客户端使用同一套工具定义，消除不一致

- 长期记忆提取管道：LLM 分析对话提取候选 → 向量去重 → LLM 冲突判断 → 落库。冲突更新走审批机制，不静默覆盖已有记忆

- 短期记忆：Redis + 磁盘双写，滑动窗口裁剪，保留完整工具调用历史以支持多轮对话

- MCP 多服务集成：Memory MCP + 高德 MCP，启动时统一连接自动发现工具。MCP 不可用时自动回退本地实现

**认知沉淀：**

工具驱动 Agent 优于规则驱动。好的 Agent 架构不是写更多的 if/else，而是把世界状态封装为工具，让模型自己推理。系统负责暴露真实世界，Agent 负责决策——这是工具自治范式的核心。


## 企业知识采集与问答双机器人系统（Dify + LangBot + 企微）

**技术栈：** Dify / 通义千问 qwen3.6-plus / 多模态视觉 LLM / LangBot / 企业微信 / 混合检索 + 重排序

基于 Dify 构建两个独立的工作流，通过 LangBot 长连接对接企业微信智能机器人。一个负责知识采集沉淀，一个负责知识问答消费，形成企业 IT 运维知识的完整闭环。

**知识采集机器人：**

将客服日常对话沉淀为结构化知识。用户发送聊天截图，通义千问多模态模型识别对话中的系统问题、根因和解决方案，自动推断所属业务系统，输出结构化 Q&A 对。提取结果先缓存，用户确认后通过 API 写入 Dify 知识库。入库前自动脱敏（身份证、手机号、工号等），支持追加修改后重新确认。

**知识问答机器人：**

问题分类器做意图识别，5 个分支覆盖不同场景。模糊咨询（未指明哪个系统）时反问用户"您要咨询的是哪个系统"，用户补充后拼接上下文二次检索，实现多轮对话中的需求澄清。对话变量缓存上一轮的系统名和问题，回答完成后自动清理。

检索策略以采集机器人沉淀的 Q&A 知识库为主、企业系统操作文档为辅，双路混合检索加重排序。LLM 基于检索上下文生成回答，禁止编造，自动拼接去重后的引用来源。

**关键设计：** 两个机器人形成知识闭环——采集机器人的输出是问答机器人的检索源。多轮上下文管理靠对话变量而非外部存储，实现轻量但可靠的状态保持。


## 企业文档 RAG 系统（数据清洗 + 检索增强 + 评测体系）

**技术栈：** Python / mineru / FAISS / BGE Embedding / BM25 + 向量混合检索 / Reranker / 通义千问

面向企业内部多格式混乱文档场景，构建完整 RAG 管线并建立评测体系。

**数据清洗管线：**
- mineru 统一处理 PDF/DOCX/PPTX/扫描件，利用视觉分析自动区分标题、正文、图片和表格，不依赖文档格式规范
- block 级输出展平为文本段落 → 语义分块（相邻句子向量相似度骤降处切开），不依赖标题层级
- 章节拆分双策略：有标题结构的按标题预切后语义合并，无结构的直接全篇分块，两种文档同一流程
- 图片通过 mineru 内置 OCR 提取文字替代，分块时通过句子→页码映射追踪 chunk 来源

**检索增强策略：**
- 混合检索：BM25 关键词 + BGE 向量语义互补，召回候选经 BGE-Reranker 精排后取 top-K
- RAG 场景选通义千问而非 DeepSeek——核心要求是忠实于上下文而非强推理，千问指令跟随更可控，不易脑补
- 检索结果注入 LLM 时附带来源文档名和页码范围，方便用户回溯原文

**评测体系（三层人工+自动结合）：**
- 离线 50 条标注测试集：每次改造检索策略后自动跑关键词覆盖回归，纯向量计算无需 LLM
- 线上自动信号：空结果率、用户点踩率、重复提问率——三个信号自动标记，超标触发人工复查
- 人工抽查：每周从线上日志随机抽 20 条，逐条验证检索质量和回答正确性，归因纠偏

关键认知：RAG 评测不存在全自动解法——召回对不对、LLM 用没用对，离开人工标注无法判断。能做的是用信号缩小人工需要看的范围，而不是消灭人工。
